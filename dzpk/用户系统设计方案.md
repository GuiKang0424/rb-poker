# 用户系统设计方案

## 📋 概述

本设计方案描述德州扑克游戏的用户注册登录系统。

**设计原则**：

- 保留零摩擦游客体验（无需注册即可游戏）
- 支持账号升级（匿名 → 正式账号）
- 筹码全局跨房间通用
- 最小可用原则，二期迭代排行榜/好友系统

---

## 🎯 核心决策总结

| 决策点 | 选择 | 说明 |
|--------|------|------|
| 需求范围 | A+D | 筹码/战绩持久化 + 为排行榜/好友系统铺路 |
| 认证方式 | D+A 混合 | 自动匿名账号 + 可升级为用户名密码 |
| 数据模型 | 方案 A（单表继承） | 一个 users 表，is_anonymous 标记 |
| 设备识别 | 方案 A | 后端生成 device_id + LocalStorage 存储 |
| Token 方案 | JWT | 无状态，30天有效期 |
| WS 认证 | 方案 B | 连接后第一条消息发送 token |
| 筹码范围 | 全局跨房间 | 单局带入不能超过账号已有筹码 |
| 持久化时机 | 每手牌结束后立即写入 | SQLite 性能足够 |
| 注册赠送 | 10000 筹码 | 新注册用户初始资金 |
| 密码加密 | bcrypt | 行业标准 |
| 邮箱验证 | 不需要 | 二期再做找回密码 |
| 排行榜 | 二期实现 | 一期只做数据持久化 |
| 数据库迁移 | 手写建表 | CREATE TABLE IF NOT EXISTS |

## 🗄️ 数据模型设计

### users 表

```sql
CREATE TABLE IF NOT EXISTS users (
    id TEXT PRIMARY KEY,           -- UUID
    username TEXT UNIQUE,          -- 用户名（匿名用户为 NULL 或 guest_xxx）
    password_hash TEXT,            -- 密码哈希（匿名用户为 NULL）
    device_id TEXT,                -- 设备ID，用于绑定/找回
    chips INTEGER DEFAULT 10000,   -- 当前筹码（注册赠送10000）
    total_hands INTEGER DEFAULT 0, -- 总手牌数
    total_wins INTEGER DEFAULT 0,  -- 赢的次数
    total_profit INTEGER DEFAULT 0,-- 总盈利
    is_anonymous BOOLEAN DEFAULT 1,-- 是否匿名账号
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_login_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
)
```

### Player 类变更（app/game/player.py）

```python
class Player:
    user_id: str | None  # 新增：关联 users.id，匿名玩家为 None
    # ... 其他字段不变（seat, chips, status, is_ready, etc.）
```

## 🔌 API 接口设计

### 1. 初始化匿名账号

```
POST /api/auth/guest
Request: { device_id: string }
Response: { user_id: string, token: string, chips: number }
```

- 前端首次访问，无 LocalStorage 时调用
- 后端生成匿名用户，返回 JWT token

### 2. 升级正式账号

```
POST /api/auth/upgrade
Auth: Bearer <token>
Request: { username: string, password: string }
Response: { success: boolean, user: UserInfo }
```

- 匿名用户设置用户名密码，升级为正式账号
- username 唯一性校验
- is_anonymous 设为 false

### 3. 登录

```
POST /api/auth/login
Request: { username: string, password: string }
Response: { token: string, user: UserInfo }
```

- 正式用户登录
- bcrypt 校验密码
- 更新 last_login_at

### 4. 获取当前用户信息

```
GET /api/auth/me
Auth: Bearer <token>
Response: UserInfo
```

```typescript
interface UserInfo {
    id: string;
    username: string | null;
    chips: number;
    isAnonymous: boolean;
    totalHands: number;
    totalWins: number;
    totalProfit: number;
    createdAt: string;
}
```

## 🔐 WebSocket 认证流程

### 连接建立后认证

```
Client → Server (连接后立即发送):
{
    type: "auth",
    token: "eyJhbGciOiJIUzI1NiIs..."
}

Server → Client (认证成功):
{
    event: "auth_success",
    user: UserInfo
}

Server → Client (认证失败):
{
    event: "auth_failed",
    reason: "invalid_token" | "expired"
}
(然后关闭连接)
```

**后端逻辑**：
- WebSocket 连接建立后，维护 `authenticated: boolean` 状态
- 未认证前只处理 `auth` 类型消息，其他消息忽略
- 认证失败（token 无效/过期）直接关闭连接
- 认证成功后，将 user_id 绑定到连接对象

### 入座筹码校验

玩家点击"准备入座"时，后端校验：
```python
if player.user_id and player.chips > user.chips:
    # 带入筹码不能超过账号已有筹码
    return error("带入筹码不能超过账号余额")
```

### 牌局结束数据更新

每手牌结束（hand_end 事件）后：
```python
for each 有 user_id 的玩家:
    user = db.query(User).get(user_id)
    user.chips = player.chips  # 更新筹码
    user.total_hands += 1      # 总手牌数 +1
    if 玩家是赢家:
        user.total_wins += 1   # 赢的次数 +1
    user.total_profit += (player.chips - initial_chips_this_hand)
    db.commit()
```

## 🔒 安全考虑

1. **JWT**：使用 HS256 算法，SECRET_KEY 从环境变量读取
2. **密码**：bcrypt hash 存储，cost factor = 12
3. **输入校验**：username 长度限制（3-20字符），正则校验
4. **SQL 注入**：全部使用 SQLAlchemy 参数化查询
5. **筹码校验**：所有筹码变更必须在后端校验，不相信前端数据

## 📝 备注

- Token 有效期：30 天（`exp` claim）
- 匿名账号清缓存即丢失，需通过登录找回正式账号
- 一期不实现：找回密码、邮箱验证、绑定手机号、好友系统
